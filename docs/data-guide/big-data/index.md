## <a name="lambda-architecture"></a><span data-ttu-id="bf462-169">Arquitetura lambda</span><span class="sxs-lookup"><span data-stu-id="bf462-169">Lambda architecture</span></span>

<span data-ttu-id="bf462-170">Ao trabalhar com conjuntos de dados muito grandes, pode levar muito tempo para executar a classificação de consultas de que os clientes precisam.</span><span class="sxs-lookup"><span data-stu-id="bf462-170">When working with very large data sets, it can take a long time to run the sort of queries that clients need.</span></span> <span data-ttu-id="bf462-171">Essas consultas não podem ser executadas em tempo real e geralmente exigem algoritmos como [MapReduce](https://en.wikipedia.org/wiki/MapReduce), que operam em paralelo em todo o conjunto de dados.</span><span class="sxs-lookup"><span data-stu-id="bf462-171">These queries can't be performed in real time, and often require algorithms such as [MapReduce](https://en.wikipedia.org/wiki/MapReduce) that operate in parallel across the entire data set.</span></span> <span data-ttu-id="bf462-172">Os resultados são então armazenados separadamente dos dados brutos e usados para consulta.</span><span class="sxs-lookup"><span data-stu-id="bf462-172">The results are then stored separately from the raw data and used for querying.</span></span>

<span data-ttu-id="bf462-173">Uma desvantagem dessa abordagem é que ela introduz latência &mdash; se o processamento levar algumas horas, uma consulta poderá retornar resultados de várias horas atrás.</span><span class="sxs-lookup"><span data-stu-id="bf462-173">One drawback to this approach is that it introduces latency &mdash; if processing takes a few hours, a query may return results that are several hours old.</span></span> <span data-ttu-id="bf462-174">O ideal é que você obtenha alguns resultados em tempo real (talvez com alguma perda de precisão) e combine esses resultados com os resultados da análise de lote.</span><span class="sxs-lookup"><span data-stu-id="bf462-174">Ideally, you would like to get some results in real time (perhaps with some loss of accuracy), and combine these results with the results from the batch analytics.</span></span>

<span data-ttu-id="bf462-175">A **arquitetura lambda**, primeiramente proposta por Nathan Marz, resolve esse problema criando dois caminhos para o fluxo de dados.</span><span class="sxs-lookup"><span data-stu-id="bf462-175">The **lambda architecture**, first proposed by Nathan Marz, addresses this problem by creating two paths for data flow.</span></span> <span data-ttu-id="bf462-176">Todos os dados recebidos pelo sistema passam por esses dois caminhos:</span><span class="sxs-lookup"><span data-stu-id="bf462-176">All data coming into the system goes through these two paths:</span></span>

* <span data-ttu-id="bf462-177">Uma **camada de lote** (caminho frio) armazena todos os dados de entrada em sua forma bruta e executa o processamento em lotes nos dados.</span><span class="sxs-lookup"><span data-stu-id="bf462-177">A **batch layer** (cold path) stores all of the incoming data in its raw form and performs batch processing on the data.</span></span> <span data-ttu-id="bf462-178">O resultado desse processamento é armazenado como uma **exibição de lote**.</span><span class="sxs-lookup"><span data-stu-id="bf462-178">The result of this processing is stored as a **batch view**.</span></span>

* <span data-ttu-id="bf462-179">Uma **camada de velocidade** (caminho quente) analisa os dados em tempo real.</span><span class="sxs-lookup"><span data-stu-id="bf462-179">A **speed layer** (hot path) analyzes data in real time.</span></span> <span data-ttu-id="bf462-180">Essa camada foi projetada para baixa latência, em detrimento da precisão.</span><span class="sxs-lookup"><span data-stu-id="bf462-180">This layer is designed for low latency, at the expense of accuracy.</span></span>

<span data-ttu-id="bf462-181">A camada de lote alimenta uma **camada de serviço** que indexa a exibição de lote para uma consulta eficiente.</span><span class="sxs-lookup"><span data-stu-id="bf462-181">The batch layer feeds into a **serving layer** that indexes the batch view for efficient querying.</span></span> <span data-ttu-id="bf462-182">A camada de velocidade atualiza a camada de serviço com atualizações incrementais de acordo com os dados mais recentes.</span><span class="sxs-lookup"><span data-stu-id="bf462-182">The speed layer updates the serving layer with incremental updates based on the most recent data.</span></span>

![Diagrama da arquitetura lambda](./images/lambda.png)

<span data-ttu-id="bf462-184">Os dados que fluem para o caminho quente são restritos por requisitos de latência impostos pela camada de velocidade, de modo que ela possa ser processada o mais rapidamente possível.</span><span class="sxs-lookup"><span data-stu-id="bf462-184">Data that flows into the hot path is constrained by latency requirements imposed by the speed layer, so that it can be processed as quickly as possible.</span></span> <span data-ttu-id="bf462-185">Geralmente, isso exige uma desvantagem de algum nível de precisão em favor dos dados que estão prontos o mais rapidamente possível.</span><span class="sxs-lookup"><span data-stu-id="bf462-185">Often, this requires a tradeoff of some level of accuracy in favor of data that is ready as quickly as possible.</span></span> <span data-ttu-id="bf462-186">Por exemplo, considere um cenário de IoT em que um grande número de sensores de temperatura envia dados telemétricos.</span><span class="sxs-lookup"><span data-stu-id="bf462-186">For example, consider an IoT scenario where a large number of temperature sensors are sending telemetry data.</span></span> <span data-ttu-id="bf462-187">A camada de velocidade pode ser usada para processar uma janela de tempo deslizante dos dados de entrada.</span><span class="sxs-lookup"><span data-stu-id="bf462-187">The speed layer may be used to process a sliding time window of the incoming data.</span></span> 

<span data-ttu-id="bf462-188">Os dados que fluem para o caminho frio, por outro lado, não estão sujeitos aos mesmos requisitos de baixa latência.</span><span class="sxs-lookup"><span data-stu-id="bf462-188">Data flowing into the cold path, on the other hand, is not subject to the same low latency requirements.</span></span> <span data-ttu-id="bf462-189">Isso permite uma computação de alta precisão em conjuntos de dados grandes, o que pode ser muito demorado.</span><span class="sxs-lookup"><span data-stu-id="bf462-189">This allows for high accuracy computation across large data sets, which can be very time intensive.</span></span> 

<span data-ttu-id="bf462-190">Em última análise, os caminhos quente e frio convergem no aplicativo cliente de análise.</span><span class="sxs-lookup"><span data-stu-id="bf462-190">Eventually, the hot and cold paths converge at the analytics client application.</span></span> <span data-ttu-id="bf462-191">Se o cliente precisar exibir dados em tempo hábil, mas potencialmente menos precisos em tempo real, ele adquirirá seu resultado do caminho quente.</span><span class="sxs-lookup"><span data-stu-id="bf462-191">If the client needs to display timely, yet potentially less accurate data in real time, it will acquire its result from the hot path.</span></span> <span data-ttu-id="bf462-192">Caso contrário, ele selecionará resultados do caminho frio para exibir dados em menos tempo hábil, mas mais precisos.</span><span class="sxs-lookup"><span data-stu-id="bf462-192">Otherwise, it will select results from the cold path to display less timely but more accurate data.</span></span> <span data-ttu-id="bf462-193">Em outras palavras, o caminho quente contém dados para uma janela relativamente pequena de tempo, após o qual os resultados podem ser atualizados com os dados mais precisos do caminho frio.</span><span class="sxs-lookup"><span data-stu-id="bf462-193">In other words, the hot path has data for a relatively small window of time, after which the results can be updated with more accurate data from the cold path.</span></span>

## <a name="kappa-architecture"></a><span data-ttu-id="bf462-199">Arquitetura de kappa</span><span class="sxs-lookup"><span data-stu-id="bf462-199">Kappa architecture</span></span>

<span data-ttu-id="bf462-200">Uma desvantagem da arquitetura de lambda é sua complexidade.</span><span class="sxs-lookup"><span data-stu-id="bf462-200">A drawback to the lambda architecture is its complexity.</span></span> <span data-ttu-id="bf462-201">A lógica de processamento aparece em dois lugares diferentes &mdash; os caminhos frio e quente &mdash; usando estruturas diferentes.</span><span class="sxs-lookup"><span data-stu-id="bf462-201">Processing logic appears in two different places &mdash; the cold and hot paths &mdash; using different frameworks.</span></span> <span data-ttu-id="bf462-202">Isso leva a uma lógica de cálculo duplicada e a complexidade de gerenciar a arquitetura para os dois caminhos.</span><span class="sxs-lookup"><span data-stu-id="bf462-202">This leads to duplicate computation logic and the complexity of managing the architecture for both paths.</span></span>

<span data-ttu-id="bf462-203">A **arquitetura de kappa** foi proposta por Jay Kreps como uma alternativa à arquitetura de lambda.</span><span class="sxs-lookup"><span data-stu-id="bf462-203">The **kappa architecture** was proposed by Jay Kreps as an alternative to the lambda architecture.</span></span> <span data-ttu-id="bf462-204">Ela tem as mesmas metas básicas da arquitetura de lambda, mas com uma diferença importante: todos os dados fluem por um único caminho, usando um sistema de processamento de fluxo.</span><span class="sxs-lookup"><span data-stu-id="bf462-204">It has the same basic goals as the lambda architecture, but with an important distinction: All data flows through a single path, using a stream processing system.</span></span>

<span data-ttu-id="bf462-206">Há algumas semelhanças na camada de lote da arquitetura de lambda, em que os dados do evento são imutáveis e todos eles são coletados, em vez de um subconjunto.</span><span class="sxs-lookup"><span data-stu-id="bf462-206">There are some similarities to the lambda architecture's batch layer, in that the event data is immutable and all of it is collected, instead of a subset.</span></span> <span data-ttu-id="bf462-207">Os dados são ingeridos como um fluxo de eventos em um log unificado distribuído e tolerante a falhas.</span><span class="sxs-lookup"><span data-stu-id="bf462-207">The data is ingested as a stream of events into a distributed and fault tolerant unified log.</span></span> <span data-ttu-id="bf462-208">Esses eventos são ordenados e o estado atual de um evento é alterado somente por um novo evento que está sendo acrescentado.</span><span class="sxs-lookup"><span data-stu-id="bf462-208">These events are ordered, and the current state of an event is changed only by a new event being appended.</span></span> <span data-ttu-id="bf462-209">Semelhante à camada de velocidade da arquitetura de uma lambda, todo o processamento de eventos é feito no fluxo de entrada e persistido como uma exibição em tempo real.</span><span class="sxs-lookup"><span data-stu-id="bf462-209">Similar to a lambda architecture's speed layer, all event processing is performed on the input stream and persisted as a real-time view.</span></span>
